Сценарий 1: Добавление дохода
Название теста: Проверка добавления новой доходной транзакции

Цель тестирования: Убедиться, что пользователь может добавить доход 
с указанием суммы, категории и описания.

Предпосылки:
- Пользователь зарегистрирован
- Создана категория для доходов
- API сервер запущен

Шаги тестирования:
1. Отправить POST запрос на /api/transactions с данными дохода
   Ожидаемый результат: Статус 201 (Created)

2. Проверить тело ответа
   Ожидаемый результат: amount = 5000, type = "income", description = "Бонус"

3. Проверить внутренние коллекции
   Ожидаемый результат: Транзакция добавлена в transactions и в список пользователя

Сценарий 2: Добавление расхода
Название теста: Проверка добавления новой расходной транзакции

Цель тестирования: Убедиться, что пользователь может добавить расход.

Предпосылки:
- Пользователь зарегистрирован
- Создана категория для расходов

Шаги тестирования:
1. Отправить POST запрос на /api/transactions с данными расхода
   Ожидаемый результат: Статус 201

2. Проверить данные
   Ожидаемый результат: amount = 1500, type = "expense"

3. Проверить добавление в коллекцию
   Ожидаемый результат: Транзакция существует в transactions

Сценарий 3: Удаление транзакции
Название теста: Проверка удаления транзакции

Цель тестирования: Убедиться, что транзакция удаляется из всех коллекций.

Предпосылки:
- Существует транзакция
- Известен ID транзакции

Шаги тестирования:
1. Проверить наличие до удаления
   Ожидаемый результат: Транзакция есть в коллекциях

2. Отправить DELETE запрос на /api/transactions/{id}
   Ожидаемый результат: Статус 200

3. Проверить отсутствие после удаления
   Ожидаемый результат: Транзакция удалена из всех коллекций

Сценарий 4: Получение статистики
Название теста: Проверка расчета общей суммы доходов и расходов

Цель тестирования: Убедиться, что статистика правильно суммирует транзакции.

Предпосылки:
- У пользователя есть несколько доходов и расходов

Шаги тестирования:
1. Добавить доходы: 1000 + 2000 = 3000
2. Добавить расходы: 500 + 300 = 800
3. Отправить GET запрос на /api/users/{id}/stats
   Ожидаемый результат: Статус 200

4. Проверить статистику
   Ожидаемый результат: total_income = 3000, total_expense = 800, balance = 2200

Сценарий 5: Фильтрация транзакций по типу
Название теста: Проверка фильтрации транзакций

Цель тестирования: Убедиться, что можно получить только доходы или только расходы.

Предпосылки:
- У пользователя есть и доходы, и расходы

Шаги тестирования:
1. Отправить GET запрос на /api/transactions?type=expense
   Ожидаемый результат: Только расходные транзакции

2. Проверить тип каждой транзакции
   Ожидаемый результат: У всех type = "expense"

Сценарий 6: Ошибка при несоответствии категории
Название теста: Проверка валидации типа категории

Цель тестирования: Убедиться, что нельзя создать доход в категории расходов.

Предпосылки:
- Есть категория расходов

Шаги тестирования:
1. Отправить POST запрос с type="income" и category_id категории расходов
   Ожидаемый результат: Статус 400

2. Проверить сообщение об ошибке
   Ожидаемый результат: "Category type mismatch"

Сценарий 7: Статистика по категориям
Название теста: Проверка группировки расходов по категориям

Цель тестирования: Убедиться, что статистика показывает разбивку по категориям.

Предпосылки:
- Есть несколько категорий с транзакциями

Шаги тестирования:
1. Добавить расходы в разные категории
2. Получить статистику
   Ожидаемый результат: В ответе есть поле by_category

3. Проверить суммы по категориям
   Ожидаемый результат: Правильные суммы для каждой категории

Сценарий 8: Удаление пользователя
Название теста: Проверка каскадного удаления транзакций

Цель тестирования: Убедиться, что при удалении пользователя удаляются все его транзакции.

Предпосылки:
- Существует пользователь с транзакциями

Шаги тестирования:
1. Проверить наличие пользователя и транзакций
2. Удалить пользователя
   Ожидаемый результат: Статус 200

3. Проверить коллекции после удаления
   Ожидаемый результат: Пользователь и все его транзакции удалены
