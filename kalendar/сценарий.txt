Сценарий 1: Создание нового события
Название теста: Проверка создания события в календаре

Цель тестирования: Убедиться, что пользователь может создать событие с датой и напоминанием.

Предпосылки:
- Пользователь зарегистрирован
- API сервер запущен

Шаги тестирования:
1. Отправить POST запрос на /api/events с данными события
   Ожидаемый результат: Статус 201 (Created)

2. Проверить тело ответа
   Ожидаемый результат: title = "Meeting", remind = 15, notified = False

3. Проверить внутренние коллекции
   Ожидаемый результат: Событие добавлено в events и в список пользователя


Сценарий 2: Получение информации о событии
Название теста: Проверка получения данных события

   Ожидаемый результат: Статус 200

   Ожидаемый результат: title = "Test Event", desc = "Description"

Сценарий 5: Фильтрация предстоящих событий
Название теста: Проверка получения только будущих событий

Цель тестирования: Убедиться, что API правильно фильтрует события по времени.

Предпосылки:
- Есть события в прошлом и будущем

Шаги тестирования:
1. Создать событие в прошлом
2. Создать событие в будущем
3. Отправить GET запрос с параметром upcoming=true
   Ожидаемый результат: Статус 200

4. Проверить список событий
   Ожидаемый результат: Только будущие события, прошлые отсутствуют


Сценарий 6: Логика создания уведомлений
Название теста: Проверка автоматического создания уведомлений

Цель тестирования: Убедиться, что система создает уведомления для подходящих событий.

Предпосылки:
- Есть события с разными настройками напоминания

Шаги тестирования:
1. Создать событие, которое должно уведомить (в прошлом)
2. Создать событие, которое уведомит сейчас
3. Вызвать проверку уведомлений POST /api/notifications
   Ожидаемый результат: count = 2

4. Проверить созданные уведомления
   Ожидаемый результат: В коллекции 2 уведомления

5. Повторно вызвать проверку
   Ожидаемый результат: Новые уведомления не создаются


Сценарий 7: Получение уведомлений пользователя
Название теста: Проверка получения уведомлений

Цель тестирования: Убедиться, что пользователь видит свои уведомления.

Предпосылки:
- У пользователя есть уведомления

Шаги тестирования:
1. Создать несколько уведомлений
2. Получить все уведомления
   Ожидаемый результат: total = количество созданных

3. Отметить одно как прочитанное
4. Получить только непрочитанные
   Ожидаемый результат: total уменьшился


Сценарий 8: Отметка уведомления как прочитанного
Название теста: Проверка изменения статуса уведомления

Цель тестирования: Убедиться, что пользователь может отметить уведомление как прочитанное.

Предпосылки:
- Существует непрочитанное уведомление

Шаги тестирования:
1. Проверить статус до отметки
   Ожидаемый результат: is_read = False

2. Отправить PUT запрос на /api/notifications/{id}
   Ожидаемый результат: Статус 200

3. Проверить обновленный статус
   Ожидаемый результат: is_read = True
